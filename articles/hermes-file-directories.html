Hermes 明明跑完了，文件却找不到：3 个目录没弄清，越用越乱


> #探寻人工智能，人与AI全新序章#


有一种情况特别让人头疼。


Hermes 明明说任务已经跑完了，甚至还告诉你"文件已经生成""结果已经保存"，可你去桌面、下载目录、项目文件夹里翻半天，什么都找不到。


这时候很多人第一反应是：是不是 Agent 又幻觉了？


有可能。


但我这段时间更常见的情况是：文件不是没生成，而是写到另一个目录去了。


尤其是你同时用 CLI（命令行）、Desktop（桌面端）、Gateway（网关）、Profile（配置档案）、Docker（容器）或者 WSL（Windows Linux 子系统）的时候，目录关系很容易乱。


这篇就讲一个很实用的问题：Hermes 跑完任务以后，文件到底可能在哪里。



先说结论：文件不一定丢了，先别急着重做


很多人找不到文件，会马上让 Hermes 重新生成一遍。


这一步先别急。


因为如果目录没弄清，重新生成一次，只是把同样的文件又写到同样找不到的地方。轻则浪费时间，重则像上一篇说的那样，把硬盘越堆越满。


我的建议是，先查 3 个目录。


第一，你启动 Hermes 时所在的目录。


第二，Hermes 网关或后台任务使用的 terminal.cwd。


第三，HOME、Profile、容器或 WSL 里的真实目录。


这 3 个目录不一定是同一个地方。


你以为它在桌面工作，它可能在项目目录。你以为它在 Windows 的 D:\work，它可能在 WSL 的 /home/user/work。你以为换了 Profile 就换了文件空间，其实它可能还在用同一个 HOME。


目录一乱，后面的判断全都会乱。



第一个目录：你启动 Hermes 的地方


如果你是在命令行里运行 Hermes，最先要看的就是当前启动目录。


比如你在这个目录启动：




cd ~/projects/demo


hermes






那很多文件就可能出现在 ~/projects/demo 下面，而不是桌面，也不是下载目录。




所以第一条命令很简单：






pwd






它会告诉你当前终端到底在哪。




然后看最近改动过的文件：






find . -maxdepth 3 -type f -mtime -1 -print






如果文件很多，可以按修改时间看：






find . -maxdepth 3 -type f -mtime -1 -print | head -50






Windows PowerShell 可以这样看当前位置：






Get-Location






再看最近一天修改过的文件：






Get-ChildItem -Recurse -File | Where-Object {$_.LastWriteTime -gt (Get-Date).AddDays(-1)} | Select-Object FullName, LastWriteTime






这一步的作用不是炫命令，而是先确认：Hermes 到底有没有在你以为的地方干活。




很多文件找不到，第一关就卡在这里。






第二个目录：terminal.cwd 影响后台任务




这里是最容易被忽略的地方。




你在命令行里启动 Hermes，和你通过网关、定时任务、消息入口让 Hermes 干活，不一定使用同一个工作目录。




Hermes 官方配置里有一个 terminal.cwd，它可以指定 Gateway（网关）和 cron（定时任务）的工作目录。也就是说，你在聊天窗口里让 Hermes 生成文件，它不一定会写到你刚才打开终端的地方，而可能写到配置里的 terminal.cwd。




配置大概长这样：






terminal:


cwd: /home/myuser/projects






如果你设置了这个路径，后台任务就可能从这里开始干活。




如果你没设置，它可能回到用户主目录附近。于是你就会遇到一种很典型的情况：




    

  • 命令行里能看到文件。
    • 

  • 桌面端里找不到。
    • 

  • 定时任务生成的东西又在另一个目录。
    • 

  • 消息网关里问它，它说保存了，但你翻项目目录没有。
    • 





这不是玄学，是工作目录没统一。




可以先打开配置文件看一眼：






nano ~/.hermes/config.yaml






或者只查 terminal 这一段：






sed -n '/terminal:/,/^[^ ]/p' ~/.hermes/config.yaml






Windows 可以打开：






notepad %USERPROFILE%\.hermes\config.yaml






我的建议是，如果你经常让 Hermes 处理文件，最好给它固定一个明确目录，比如：






terminal:


cwd: /home/myuser/hermes-work






不要今天让它在桌面跑，明天让它在下载目录跑，后天又让它在某个项目目录跑。人都记不住，Agent 更容易乱。






第三个目录：HOME、Profile 和下载目录不是一回事




很多人会把 Profile 理解成一个完整小号。




这句话只对一半。




Profile 可以隔离一部分 Hermes 自己的配置、记忆、会话和任务状态，但它不等于完整沙箱，也不等于自动给你换了一个系统 HOME。




更直接一点说：




    

  • 你换了 Profile，不代表文件一定换地方。
    • 

  • 你换了 Profile，不代表 GitHub、SSH、npm 这些工具都换账号。
    • 

  • 你换了 Profile，也不代表下载目录、缓存目录、临时目录完全隔离。
    • 





Hermes 配置里还有 home_mode，它会影响子进程看到的 HOME。默认情况下，本机上的很多外部命令可能还是沿用你真实系统用户的 HOME，这样 git、ssh、gh、npm 这些工具比较容易正常工作。




好处是少折腾。




坏处是你以为隔离了，其实某些文件和账号还在共用。




所以排查文件时，除了看当前目录，还要看 HOME：






echo $HOME






再看 Hermes 相关目录：






ls -la ~/.hermes






如果你怀疑文件写到了 HOME 下面，可以查最近文件：






find ~ -maxdepth 4 -type f -mtime -1 -print | head -100






Windows PowerShell 可以查用户目录：






$env:USERPROFILE


Get-ChildItem $env:USERPROFILE -Recurse -File -ErrorAction SilentlyContinue | Where-Object {$_.LastWriteTime -gt (Get-Date).AddDays(-1)} | Select-Object FullName, LastWriteTime






这一步非常适合排查"我明明在项目里找不到，但它说已经生成了"的情况。






Docker 和 WSL 最容易路径错位




如果你用 Docker、WSL 或远程机器，文件找不到的概率会更高。




原因很简单：你看到的路径，和 Agent 真正执行命令的路径，可能不是同一个系统。




比如 Windows 上的路径可能是：






D:\ai\hermes-work






WSL 里看到的可能是：






/mnt/d/ai/hermes-work






Docker 容器里看到的又可能是：






/workspace






你让 Hermes "保存到桌面"，它未必知道你说的是 Windows 桌面、WSL 桌面，还是容器里的某个目录。




所以这类环境里，最稳的做法不是口头说"保存到桌面"，而是给绝对路径。




比如：




> 请把结果保存到 /mnt/d/ai/hermes-output 目录。


> 保存完成后，运行 pwd 和 ls -la /mnt/d/ai/hermes-output，把输出贴给我。




如果是 Docker，也要确认挂载目录。




不要只问：




> 文件保存好了吗？




要问：




> 请告诉我你当前的 pwd。


> 请列出你刚才写入的完整路径。


> 请运行 ls -la 检查文件是否真的存在。




这几句话很朴素，但能减少很多"它说保存了，我却找不到"的情况。






最小排查流程：先跑这 4 条




如果你现在就遇到文件找不到，可以按这个顺序查。




**第一，确认当前目录：**






pwd






**第二，查当前目录最近生成的文件：**






find . -maxdepth 4 -type f -mtime -1 -print | head -100






**第三，查 HOME 下面最近生成的文件：**






find ~ -maxdepth 4 -type f -mtime -1 -print | head -100






**第四，看 Hermes 日志：**






hermes logs -n 100






如果怀疑是网关或后台任务写错目录，再看：






hermes logs gateway -n 100






日志里不一定直接告诉你文件在哪，但它常常能暴露几个关键信号：




    

  • 任务从哪个目录启动。
    • 

  • 工具有没有真正执行。
    • 

  • 命令有没有报错。
    • 

  • 是不是写到了临时目录。
    • 

  • 是不是 Gateway 跟 CLI 用的上下文不同。
    • 





找文件不要只靠回忆。




先看目录，再看日志，再让 Hermes 复述完整路径。






我的建议：给 Hermes 固定一个输出目录




用久了以后，我越来越不建议让 Hermes 随便写文件。




最稳的做法是给它一个固定输出目录。




比如：






mkdir -p ~/hermes-output






然后以后所有任务都加一句：




> 所有新生成的文件都保存到 ~/hermes-output。


> 不要保存到桌面、下载目录或临时目录。


> 保存完成后，列出完整文件路径和文件大小。




如果你是 Windows 用户，也可以固定到一个清楚的位置：




> 所有新生成的文件都保存到 D:\hermes-output。


> 保存完成后，列出完整文件路径和文件大小。




如果你用 WSL，就把 Windows 路径换成 WSL 能识别的路径：




> 所有新生成的文件都保存到 /mnt/d/hermes-output。


> 保存完成后，运行 ls -la /mnt/d/hermes-output。




这一步的价值很大：




1. 你不用到处翻文件。


2. Agent 不容易把结果写到临时目录。


3. 后续清理也方便。


4. 遇到问题可以直接围绕一个目录排查。




如果你愿意更稳一点，还可以让 Hermes 每次任务结束后做一次确认：




> 任务结束前，请做 3 件事：


> 1. 告诉我最终文件保存在哪里。


> 2. 运行命令确认文件确实存在。


> 3. 告诉我文件大小和最后修改时间。




这比单纯让它说"已完成"靠谱得多。






干货提炼




Hermes 文件找不到，很多时候不是文件丢了，而是目录没弄清。




先查启动目录，再查 terminal.cwd，最后查 HOME、Profile、Docker/WSL 的真实路径。




不要急着重做，先跑 4 条排查命令。




如果想让 Agent 更靠谱，给它固定一个输出目录，每次任务结束后做三连确认。




> 目录清楚了，文件就不会丢。
